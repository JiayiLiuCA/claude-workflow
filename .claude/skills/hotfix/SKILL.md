你现在处于 Hotfix 快速通道。参数即本次改动的描述。目标：以最小仪式完成一个小改动，同时不让它逃出体系的记忆——体系外的静默改动会让 PROGRESS 与契约索引失真。

# 适用判据（先自检，不满足则停）

全部满足才可走本通道：

1. 改动小而局部（量级参考：≤3 个文件、≤50 行）
2. 不改 API 契约、不改 DB schema、不新增依赖、不改架构决策
3. 不属于任何进行中 step 的范围（属于的话应在该 step 内做）

任何一条不满足：停下，告诉用户这需要走正式 step（`plan step N`）并说明理由。

# 流程

1. **同步 main**：`git fetch origin`；确认在 main 且与 origin/main 一致。当前分支有未提交工作时先停下向用户确认（stash 或先完成）。若 main 受保护或用户要求，改为从 origin/main 建 `fix/<slug>` 分支走 PR。
2. **改动**：完成修改，跑受影响的测试 / 类型检查；无自动化覆盖时手动验证并说明验证方式。
3. **记录（分级）**：
   - 改动影响行为（用户可感知 / 修了真 bug）：在 `docs/planning/PROGRESS.md` 顶部「杂项（hotfix log）」节加一行：`- {日期} \`{commit 短 hash}\` — {一句话描述}`
   - 纯 typo / 注释 / 依赖 bump：不动 PROGRESS，commit message 即记录
4. **commit**：`Hotfix: <描述>`；直接推 main（或走 PR，见第 1 步）。
5. **汇报**：改了什么、如何验证、是否记入 PROGRESS。

# 禁止事项

- 不顺手做描述之外的任何改动——发现别的问题：报告，不修
- 不把多个不相关小修打包成一个 hotfix——一事一个
- 本通道不设阶段标记（hook 不限制写入范围），纪律全靠上述判据自律
